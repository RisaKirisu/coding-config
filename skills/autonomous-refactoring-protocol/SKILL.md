Goal: refactor the target modules into readable, maintenable, and elegant code bases. Concise code, verbose comment. Public interfaces MUST NOT change.

## Refactor autonomously in a loop
Refactor loop:
0. If a compression just happened, rebuild context, including this skill and other loaded skills.
1. Load skills TDD, improve-codebase-architecture, ponytail[full] skill, and other relevant skills.
2. Follow improve-codebase-architecture to explore improvement opportunity. Ignore the skill's inst to generate HTML. Only generate MD report for self-reference.
3. Launch 2 general agents to independently validate the identified refactor suggestions -- do they truly improve the codebase's readability and maintenability, or are they noise generated by LLM's tendency to always output something.
4. Plan for impl of validated refactor suggestions. Write one issue for each refactor work, and persist them to `.scratch/<task-module>/refactor-issues`.
5. Compress the previous steps into concise summary -- if previous summaries exist, compress them as well.

6. For each issue, stop after all refactor issues are completed:
    1. Plan and impl the refactor issue.
    2. Compress, load skill review and ponytail-review, and perform single review pass to ensure public behavior stay unchanged and refactor goal is met. Fix any logic defect.
    3. Mark the issue as complete, compress and start the next issue.

7. Compress the session into concise summary -- if previous summaries exist, compress them as well.
8. Repeat from step 1.

[StopCondition]
When step 3 returned result and your own review determines that the module codebase is high-quality, maintenable, and highly readable.

## Important Notes:
- Work autonomously.
- Create progress tracker in `.scratch/autonomous-progress/- <task>-progress.md`
- Append progress entry with iso format local time after each major step.
- Each entry is short, current progress only. No detailed notes.
- Launch `explore` subagents to perform code, reference, or docs - researches in parallel. They can involve web search, context7, or repo research.
- When performing all compression and compaction, always list the context and skills to reload in the compaction summary. 

## Helper-specific rules:
- A private helper must earn its existence. Add one only when it encodes a real invariant or rule, hides a meaningful boundary, reduces repeated logic across several call sites, separates branching or state changes that would make the caller hard to follow, or gives a domain concept a name stronger than the inline code itself.
- Avoid helpers that only wrap one expression, forward arguments unchanged, serve one caller while forcing a reader to jump away from local control flow, rename obvious code without reducing complexity, or split one linear operation into a chain of tiny private functions.
- When a helper is reused fewer than three times and contains little logic or branching, prefer inline code unless the helper protects a real invariant or boundary. For example, a one-use helper like `extract_forward_expansion_json(response)` that only returns `response.messages.or(response.message)` is usually clearer as inline code at the protocol response handling site.
